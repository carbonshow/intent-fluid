# Trace Protocol Specification

> Version 1.0 — the canonical standard for execution tracing in intent-fluid skills.

## 1. Overview

The trace protocol provides a framework-level mechanism for skills to emit structured execution events. These events power:

- **Real-time visualization** via the built-in web dashboard
- **Static export** to Mermaid DAGs, markdown summaries, and ASCII timelines
- **External observability** via JSONL ingestion into tools like Datadog, Grafana Loki, or `jq` pipelines

The protocol is skill-agnostic. Any skill can emit trace events by calling `scripts/trace.sh` and declaring its workflow structure in the `SKILL.md` frontmatter.

## 2. Trace File

Each task produces a single trace file:

```
<task_dir>/trace.jsonl
```

Format: **JSONL** (JSON Lines) — one JSON object per line, UTF-8 encoded, appended incrementally.

The file is created during task initialization (e.g., by the skill's `init.sh` or equivalent setup step). It must exist before the first `trace.sh` call.

## 3. Event Schema

Each line in `trace.jsonl` is a JSON object conforming to this schema:

```json
{
  "id": "evt_001",
  "ts": "2026-03-27T18:00:20Z",
  "skill": "surge",
  "type": "step_start",
  "step": "analyze",
  "round": 1,
  "agent": "director",
  "status": "executing",
  "status_display": "⚡ Analyzing requirements",
  "detail": {},
  "parent_id": null,
  "tags": []
}
```

### Field Reference

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | Auto-generated unique event ID. Format: `evt_{NNN}` (zero-padded 3+ digits, auto-incremented from file line count). |
| `ts` | string | yes | ISO 8601 timestamp (UTC). Auto-generated by `trace.sh`. |
| `skill` | string | yes | Name of the skill emitting the event. Must match the skill's `name` frontmatter field. |
| `type` | string | yes | Event type. See §4 Event Types. |
| `step` | string | yes | Current execution step name. Must be one of the steps declared in the skill's `trace.steps` frontmatter. |
| `round` | number | yes | Current execution round (1-indexed). For `linear` topology skills, always `1`. For `cyclic` skills, increments with each iteration. |
| `agent` | string | yes | Agent identifier. `"director"` for the orchestrator; `"subagent:<role>"` for dispatched agents. |
| `status` | string | yes | Machine-readable status. Common values: `"executing"`, `"completed"`, `"failed"`, `"validating"`, `"retrying"`. |
| `status_display` | string | yes | Human-readable status line with emoji prefix. Displayed in the dashboard status bar and terminal output. |
| `detail` | object | no | Arbitrary key-value metadata. See §5 Detail Object. |
| `parent_id` | string \| null | no | ID of the parent event, forming a tree. See §6 Event Tree. |
| `tags` | string[] | no | Skill-specific tags in `key:value` format. Allows skills to attach domain-specific metadata without polluting the core schema. |

## 4. Event Types

| Type | Semantics | Typical Emitter |
|------|-----------|-----------------|
| `step_start` | A workflow step is beginning. | Director, before dispatching work. |
| `step_end` | A workflow step has completed (successfully or not). | Director, after validation passes. |
| `agent_dispatch` | A subagent is being dispatched for a task. | Director, at dispatch time. |
| `agent_return` | A subagent has returned its result. | Director, when subagent completes. |
| `checkpoint` | A user-facing checkpoint or gate has been reached. | Director, at decision points. |
| `decision` | A routing or convergence decision has been made. | Director, after evaluating results. |
| `error` | An error occurred (truncation, permission, validation failure). | Director, on error detection. |

### Event Type Lifecycle

A typical step lifecycle emits events in this order:

```
step_start → agent_dispatch → agent_return → step_end
                                   ↑
                              (or: error → retry → agent_dispatch → agent_return → step_end)
```

For Director-orchestrated steps (no single subagent), `agent_dispatch`/`agent_return` may repeat multiple times within a single `step_start`/`step_end` pair.

## 5. Detail Object

The `detail` field carries step-specific metadata. Common keys:

| Key | Type | Description |
|-----|------|-------------|
| `input_files` | string[] | Files read as input for this step. |
| `output_files` | string[] | Files produced by this step. |
| `validation_result` | string | Output validation classification: `"PASS"`, `"MINOR_TRUNCATION"`, `"SEVERE_TRUNCATION"`. |
| `decision` | string | The decision made (e.g., `"continue"`, `"converge"`, `"rollback:L2"`). |
| `error_message` | string | Description of the error (for `error` events). |
| `duration_ms` | number | Approximate duration in milliseconds (if measurable). |

Skills may add additional keys as needed. Unknown keys are preserved and displayed in the dashboard detail panel.

## 6. Event Tree (parent_id)

Events form a tree via `parent_id`:

```
null (root)
├── evt_001 (step_start: analyze, round 1)
│   ├── evt_002 (agent_dispatch: subagent:analyze)
│   └── evt_003 (agent_return: subagent:analyze)
├── evt_004 (step_end: analyze, round 1)
├── evt_005 (step_start: design, round 1)
│   ├── evt_006 (agent_dispatch: subagent:expert_backend_architect)
│   ├── evt_007 (agent_dispatch: subagent:expert_security)
│   ├── evt_008 (agent_return: subagent:expert_backend_architect)
│   ├── evt_009 (agent_return: subagent:expert_security)
│   └── evt_010 (checkpoint: design_confirmed)
└── evt_011 (step_end: design, round 1)
```

**Rules:**
- `step_start` and `step_end` events have `parent_id: null` (top-level).
- `agent_dispatch`, `agent_return`, `checkpoint`, `decision`, and `error` events should reference their enclosing `step_start` event's `id` as `parent_id`.
- The tree structure enables the dashboard to show nested detail (expand a step to see its subagents).

## 7. Tags Convention

Tags allow skills to attach domain-specific metadata without extending the core schema.

**Format:** `key:value` strings.

**Examples (surge):**
- `iteration_type:full`
- `iteration_type:lightweight`
- `eval_level:L1+L2`
- `deviation_level:Level_2`
- `convergence:plateau`

**Rules:**
- Keys use `snake_case`.
- Values are free-form strings (no spaces; use underscores).
- Tags are optional. The dashboard displays them as badges on event nodes.
- Do not duplicate core schema fields in tags.

## 8. Workflow Declaration (SKILL.md Frontmatter)

Skills declare their workflow structure in the `trace` frontmatter field. See `docs/SKILL_SPEC.md` §5.2 for the full specification.

```yaml
---
name: my-skill
description: "Use when ..."
trace:
  steps: [step1, step2, step3]
  topology: linear
  max_rounds: 1
---
```

| Subfield | Required | Type | Description |
|----------|----------|------|-------------|
| `steps` | yes | string[] | Ordered list of step names in the workflow. |
| `topology` | yes | string | `linear` (single pass), `cyclic` (iterative loop), or `dag` (general directed acyclic graph). |
| `max_rounds` | no | number | Maximum rounds for `cyclic` topology. Default: `1` for `linear`, `5` for `cyclic`. |

The dashboard reads this declaration to select its layout strategy:
- **linear** → vertical chain
- **cyclic** → step × round grid with cross-round edges
- **dag** → topological sort with layered layout

## 9. Framework Scripts

### `scripts/trace.sh` — Emit a Trace Event

```bash
bash scripts/trace.sh <trace_file> <skill> <type> <step> <round> <agent> [detail_json]
```

**Arguments:**

| Arg | Required | Description |
|-----|----------|-------------|
| `trace_file` | yes | Path to the `trace.jsonl` file. |
| `skill` | yes | Skill name (e.g., `surge`). |
| `type` | yes | Event type (see §4). |
| `step` | yes | Current step name. |
| `round` | yes | Current round number. |
| `agent` | yes | Agent identifier. |
| `detail_json` | no | JSON string for the `detail` field. Default: `{}`. |

The script auto-generates `id` and `ts`. It appends one JSONL line atomically.

**Tags:** Pass tags within the `detail_json` under a `"tags"` key:
```bash
bash scripts/trace.sh ./trace.jsonl surge step_start analyze 1 director \
  '{"input_files":["context.md"],"tags":["iteration_type:full","eval_level:L1"]}'
```

### `scripts/trace-export.sh` — Export Trace to Visualization Formats

```bash
bash scripts/trace-export.sh <trace_file> <format> [--skill-dir <path>]
```

Formats: `mermaid`, `summary`, `timeline`.

### `scripts/dashboard.sh` — Real-time Web Dashboard

```bash
bash scripts/dashboard.sh start <task_dir> [--skill-dir <path>]
bash scripts/dashboard.sh stop  <task_dir>
bash scripts/dashboard.sh status <task_dir>
```

See the script files for detailed usage.

## 10. Integration Guide — Adding Trace to a New Skill

### Step 1: Declare Workflow in Frontmatter

```yaml
---
name: code-review
description: "Use when ..."
trace:
  steps: [scan, review, report]
  topology: linear
  max_rounds: 1
---
```

### Step 2: Create trace.jsonl During Initialization

In your skill's setup step (or `init.sh`), create the trace file:

```bash
touch "${TASK_DIR}/trace.jsonl"
```

### Step 3: Emit Events at State Transitions

Add trace emission instructions to your `SKILL.md` body. The agent (Director or equivalent) calls `trace.sh` at each state transition:

```markdown
### Trace Emission Points

At each state transition, emit a trace event:

| Moment | Type | Example |
|--------|------|---------|
| Before starting a step | `step_start` | `bash scripts/trace.sh {trace_file} code-review step_start scan 1 director` |
| Dispatching a subagent | `agent_dispatch` | `bash scripts/trace.sh {trace_file} code-review agent_dispatch scan 1 subagent:scanner '{"parent_id":"evt_001"}'` |
| Subagent returned | `agent_return` | `bash scripts/trace.sh {trace_file} code-review agent_return scan 1 subagent:scanner '{"parent_id":"evt_001"}'` |
| Step completed | `step_end` | `bash scripts/trace.sh {trace_file} code-review step_end scan 1 director '{"output_files":["scan_report.md"]}'` |
```

### Step 4: (Optional) Enable Dashboard

Ask the user during startup if they want the real-time dashboard:

```bash
bash scripts/dashboard.sh start "${TASK_DIR}" --skill-dir "${SKILL_DIR}"
```

Stop the dashboard when the task completes:

```bash
bash scripts/dashboard.sh stop "${TASK_DIR}"
```

### Step 5: (Optional) Generate Static Exports

After task completion:

```bash
bash scripts/trace-export.sh "${TASK_DIR}/trace.jsonl" mermaid --skill-dir "${SKILL_DIR}"
bash scripts/trace-export.sh "${TASK_DIR}/trace.jsonl" summary --skill-dir "${SKILL_DIR}"
```

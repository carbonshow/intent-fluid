───────────────────────────────────────────────────────────────────────────────
                          NAMING CONVENTIONS
───────────────────────────────────────────────────────────────────────────────

Skill Directory Name:
  ├─ Format: [a-z0-9-]
  ├─ Length: 2-50 chars recommended
  ├─ Examples: surge, code-review, tdd-guard
  └─ MUST MATCH: name field in SKILL.md frontmatter

SKILL.md Frontmatter:
  ├─ name: [a-z0-9-] (required, must match directory)
  ├─ description: ≤1024 chars, trigger-focused (required)
  ├─ version: semver (optional, default 0.1.0)
  ├─ author: free text (optional)
  ├─ tags: discovery tags (optional)
  ├─ platforms: [claude, cursor, gemini] (optional)
  └─ trace: workflow structure for observability (optional)

Phase Output Files (Surge):
  ├─ iter_{NN}_{phase}.md            (NN = zero-padded iteration)
  ├─ iter_{NN}_implement_{module}.md (parallel modules)
  └─ iter_{NN}_expert_review_{role}.md (expert reviews)

───────────────────────────────────────────────────────────────────────────────
                          KEY DESIGN PATTERNS
───────────────────────────────────────────────────────────────────────────────

1. PROGRESSIVE DISCLOSURE
   └─ Keep SKILL.md focused (< 500 lines)
   └─ Move bulk material to references/ subdirectory
   └─ Load on-demand with explicit agent instructions
   └─ Benefit: Lower context window load, clearer progression

2. CANONICAL SPEC + VALIDATION
   └─ SKILL_SPEC.md v2.0 defines all rules
   └─ validate-skill.sh enforces compliance
   └─ Prevents drift between skills
   └─ Easy onboarding for new skill authors

3. INDEPENDENT SKILL VERSIONING
   └─ Each skill has own semver in SKILL.md
   └─ Repository version bumps for spec changes, new skills
   └─ Allows rapid iteration on individual skills

4. STRUCTURED EXECUTION TRACING
   └─ trace.jsonl (JSONL format, append mode)
   └─ Supports real-time dashboard, static export, external observability
   └─ Event tree structure (parent_id relationships)

5. CROSS-PLATFORM DISTRIBUTION
   └─ Platform-specific manifests (.claude-plugin, .cursor-plugin, etc.)
   └─ Common source: skills/ directory with standardized SKILL.md
   └─ Installation commands per platform

6. CONTEXT PACKAGE PATTERN (Surge)
   └─ Single source of truth: {task_dir}/ contains all state
   └─ state.md maintained via scripts/state.sh (prevents corruption)
   └─ Iterations stored as iter_NN_phase.md files
   └─ Trace events logged to trace.jsonl for observability

───────────────────────────────────────────────────────────────────────────────
                          TOKEN BUDGET BEST PRACTICES
───────────────────────────────────────────────────────────────────────────────

From Surge Skill Implementation:

1. SUMMARIES, NOT FULL CONTENT
   └─ Pass upstream output summaries to subagents, not full documents
   └─ Significantly reduces context load

2. ROLLING CONTEXT WINDOWS
   └─ Trim older iterations if iteration ≥ 3
   └─ Keep only recent and critical context

3. RESEARCH BY REFERENCE
   └─ Link to search results rather than embed full content
   └─ Subagents can fetch on demand

4. SCOPE REDUCTION THRESHOLD
   └─ If subagent prompt > ~40% of context window (~80K chars), reduce scope
   └─ Prevents truncation and quality degradation

2. Create SKILL.md:
